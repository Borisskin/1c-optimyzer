# МоделированиеТЖ.epf

Внешняя обработка для **контролируемого моделирования событий технологического журнала** 1С (TLOCK, TDEADLOCK, DBMSSQL) с целью тестирования инструмента анализа ТЖ **1C-Optimyzer**.

**Версия платформы**: 1С:Предприятие 8.3.27 (тестировалось на 8.3.27.1859)
**Целевая конфигурация**: 1С:Бухгалтерия предприятия 3.0
**База**: Test1CProf на сервере localhost:2541 (параметры можно сменить в форме)

## Быстрый старт

1. **Установи расширение** (рекомендуется, см. ниже)
2. Запусти Предприятие на Test1CProf
3. **Файл → Открыть** → `D:\1C-Optimyzer\МоделированиеТЖ\build\МоделированиеТЖ.epf`
4. Жми кнопки сценариев → смотри таблицу «История запусков» снизу
5. После прогона — собирай ТЖ-логи и грузи в 1C-Optimyzer

## Архитектура: две стратегии запуска воркеров

Внешняя обработка **не может** запустить свой метод во `ФоновыеЗадания.Выполнить` — платформа требует общий модуль конфигурации или расширения. Поэтому реализованы две стратегии:

### Стратегия 1 (предпочтительная) — фоновые задания через расширение

Расширение `ТЖМоделированиеРасш` в папке `D:\1C-Optimyzer\ТЖМоделированиеРасш\` содержит общий модуль `ВоркерыТЖ` с экспортными методами. Обработка вызывает:

```bsl
ФоновыеЗадания.Выполнить("ВоркерыТЖ.БлокировкаHolder", Массив(ИмяРегистра, 15))
```

**Преимущества**:
- Запуск воркера ~100-300 мс (vs 3-10 сек для отдельного процесса)
- Гарантированная синхронизация (Holder успевает взять блокировку до Waiter)
- Воркеры в `rphost` — серверная лицензия, не открываются клиентские окна

### Стратегия 2 (fallback) — параллельные процессы 1cv8c.exe

Если расширение не установлено, обработка детектит это (ФЗ кидает исключение «процедура не найдена») и переключается на запуск отдельных процессов через `WScript.Shell.Run`:

```
1cv8c.exe ENTERPRISE /S<srv>\<ref> /N<user> /Execute МоделированиеТЖ.epf /C"TLOCK_HOLDER:Регистр:15:1"
```

Внутри воркер-процесса в `ПриОткрытии` детектится `ПараметрЗапуска` → форма не открывается → серверный метод выполняется → приложение закрывается.

**Когда fallback актуален**: если не можешь/не хочешь ставить расширение в базу.

## Установка расширения (один раз)

**Перед установкой закрой Конфигуратор на Test1CProf** (DESIGNER не работает параллельно).

```powershell
cd D:\1C-Optimyzer\ТЖМоделированиеРасш
.\install.ps1
```

По умолчанию: `localhost:2541\Test1CProf`, пользователь BVS без пароля. Для других параметров — `-Server`, `-Ref`, `-User`, `-Password`.

Проверка после установки: в Конфигураторе **Конфигурация → Расширения** должен быть `ТЖМоделированиеРасш` (AddOn, активно).

Удаление: `.\uninstall.ps1` из той же папки.

## Сценарии

Каждая кнопка запускает одну модель события ТЖ. Результат пишется в таблицу «История запусков». В колонке **Статус** будет `ФЗ` (фоновое задание через расширение) или `Shell` (fallback через 1cv8c.exe).

### 1. TLOCK (долгое ожидание блокировки)

- Параметр **«TLOCK событий»** (по умолчанию 10) — сколько пар воркеров запустить
- Для каждой пары: Holder берёт X на `AccumulationRegister.ОплатаСчетов` на 15 сек → пауза 1 сек → Waiter пытается взять ту же блокировку → ждёт ~14 сек → платформа пишет **TLOCK**

**Цель в ТЖ**: N штук TLOCK длительностью ~14 сек.

### 2. Дедлок типа 1 — эскалация S → X (ЦУП 2.12.3.2)

Два воркера на одном регистре `ОплатаСчетов`: транзакция → S → пауза 5 сек → пытаются повысить до X. Оба не могут — дедлок.

**Цель в ТЖ**: 1 событие `TDEADLOCK` с типом эскалации.

### 3. Дедлок типа 2 — разный порядок (ЦУП 2.12.3.3)

Воркер A: `Регистр1` → `Регистр2`; Воркер B: `Регистр2` → `Регистр1`. Взаимное ожидание → дедлок.

**Цель в ТЖ**: 1 `TDEADLOCK` типа «разный порядок».

### 4. Дедлок типа 3 — один ресурс

Оба воркера: S на вспомогательном → X на основном (`Регистр3`). Конкуренция за X.

**Цель в ТЖ**: 1 `TDEADLOCK` типа «один ресурс».

### 5. DBMSSQL (медленные запросы)

20 запросов по умолчанию, чередуются 5 антипаттернов:
1. Full table scan
2. Подзапрос в условии (ЦУП 2.13.4)
3. Группировка по большому набору полей
4. Левое соединение двух больших регистров
5. Сортировка по выражению `ВЫРАЗИТЬ`

**Цель в ТЖ**: N событий `DBMSSQL` длительностью > 100 мс (фильтр logcfg.xml).

### 6. Все подряд

Сценарии 1-5 последовательно с паузами 3-20 сек между ними.

## Требования

1. **1С:Предприятие 8.3.27** (`C:\Program Files\1cv8\8.3.27.1859\bin\1cv8c.exe`)
2. **БП 3.0** с регистрами в режиме Managed:
   - `AccumulationRegister.ОплатаСчетов`
   - `AccumulationRegister.НачисленияУдержанияПоСотрудникам`
   - `AccumulationRegister.ОценочныеОбязательства`
3. **logcfg.xml** настроен для TDEADLOCK / TLOCK (>1 сек) / DBMSSQL (>100 мс)
4. **Расширение** установлено (рекомендуется) **или** `WScript.Shell` доступен (для fallback)

## Как проверить что события попали в ТЖ

После прогона:

1. Открой папку ТЖ-логов (например `C:\Program Files\1cv8\srvinfo\reg_2541\<база-id>\rphost_*\<дата>.log`)
2. Найди файл за период прогона
3. Поищи:
   - `,TLOCK,` (Сценарий 1)
   - `,TDEADLOCK,` (Сценарии 2-4)
   - `,DBMSSQL,` с длительностью > порога (Сценарий 5)
4. Скопируй папку ТЖ-логов в архив и загрузи в **1C-Optimyzer**

В стратегии «ФЗ» события будут привязаны к `BackgroundJob` контексту; в стратегии «Shell» — к `Process` отдельных клиентов. Оба варианта валидны для тестирования 1C-Optimyzer.

## Безопасность

- **Все транзакции откатываются** — в базе ничего не сохраняется
- **Никаких движений по регистрам** — только блокировки и сразу откат
- При запуске показывается предупреждение что обработка для тестовой базы
- DBMSSQL делает только `Запрос.Выполнить()` — это чтение

> ⚠️ **Не запускай на продуктивной базе.** Даже без записи — блокировки тормозят реальную работу пользователей на ~15 сек минимум.

## Известные ограничения и места проверки

1. **Дедлок статистический** — может не случиться с первого раза. Повтори сценарий 2-4 если в ТЖ не появилось TDEADLOCK.
2. **Стратегия ФЗ требует прав на запуск фоновых заданий** — пользователь BVS должен иметь роль с правом RunBackgroundJob. У администратора по умолчанию есть.
3. **Стратегия Shell требует параллельных лицензий** — каждый воркер открывает клиент 1С. Если лицензий 1-2, сценарий 1 (10 пар = 20 сессий) упадёт. Уменьши **TLOCK событий** до 1-2 если ограничен по лицензиям.
4. **Имена регистров можно менять** в форме под любую конфигурацию с регистрами в Managed-режиме.
5. **Путь к 1cv8c.exe** проверь — должен соответствовать установленной у тебя платформе.
