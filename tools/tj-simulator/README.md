# TJ Simulator — генератор событий технологического журнала

Набор инструментов для **контролируемой генерации событий ТЖ** (TLOCK, TDEADLOCK, DBMSSQL, EXCP) на тестовой базе 1С. Используется для:

- Получения архивов ТЖ с заранее известными типами событий для тестирования 1C-Optimyzer.
- Real-data validation Deadlock Anatomy (Sprint 3 Phase D).
- Тренировки rule classifier на известных паттернах.

**Создан в Sprint 3.5** ([отчёт](../../docs/SPRINT_3_5_REPORT.md)) для разблокировки Sprint 3 follow-up — в реальном архиве Сергея не было `TDEADLOCK` и почти не было `TLOCK` событий (logcfg.xml + типовая БП 3.0 без нагрузки).

## Quick start

> **Требуется:** платформа 1С 8.3.27+ , тестовая база (например, типовая БП 3.0) с правами `DataLockControlMode = Managed` на нужных регистрах. **НЕ запускать на продакшене** — все транзакции откатываются, но создаётся нагрузка.

### 1. Открыть обработку

```
1С:Предприятие → Файл → Открыть → tools/tj-simulator/МоделированиеТЖ/build/МоделированиеТЖ.epf
```

### 2. Настроить параметры на форме

- Строка подключения: `Srvr="localhost:2541";Ref="Test1CProf"` (под свою базу)
- Пользователь / Пароль
- Регистры для блокировок (по умолчанию 3 регистра БП 3.0)
- Количество TLOCK событий / DBMSSQL запросов

### 3. Запустить нужный сценарий

| Кнопка | Что генерирует | Сколько событий |
|---|---|---|
| **1. TLOCK** | Управляемые блокировки с конкуренцией Holder ↔ Waiter | ~10 TLOCK на запуск (по КоличествоTLOCK) |
| **2. Дедлок: эскалация** | S→X эскалация в одной транзакции (ЦУП 2.12.3.2) | 1 TDEADLOCK + 1 EXCP |
| **3. Дедлок: разный порядок** | Захват регистров в разном порядке двумя сессиями (ЦУП 2.12.3.3) | 1 TDEADLOCK + 1 EXCP |
| **4. Дедлок: один ресурс** | X-X конкуренция после S | 1 TDEADLOCK + 1 EXCP |
| **5. DBMSSQL** | 20 неэффективных запросов (full scan, плохой JOIN, sort by expression) | ~20 DBMSSQL > 200 мс |
| **6. Все подряд** | 1+2+3+4+5 с паузами | ~10 TLOCK + 3 TDEADLOCK + ~30 EXCP + 20 DBMSSQL |

### 4. Собрать архив

Открыть 1C-Optimyzer → загрузить `C:\1C-TechLog` → проверить что новые события появились на странице «События ТЖ».

## Пересборка из исходников

Если меняешь `.bsl` или JSON формы — пересобери:

```powershell
# Из корня репо
cd tools/tj-simulator/МоделированиеТЖ
powershell.exe -NoProfile -File "..\..\..\..\.claude\skills\epf-build\scripts\epf-build.ps1" `
  -InfoBaseServer "localhost:2541" -InfoBaseRef "Test1CProf" -UserName "BVS" `
  -SourceFile "src/МоделированиеТЖ.xml" `
  -OutputFile "build/МоделированиеТЖ.epf"
```

**Важно:** после правки `.bsl` восстанови UTF-8 BOM:
```powershell
$path = "src/МоделированиеТЖ/Forms/Форма/Ext/Form/Module.bsl"
$content = Get-Content -Path $path -Raw -Encoding UTF8
[System.IO.File]::WriteAllText($path, $content, (New-Object System.Text.UTF8Encoding($true)))
```

Без BOM 1С молча теряет экспортные методы.

## Архитектурные решения (зафиксированы в Sprint 3.5)

- **Параллелизм воркеров** — через `1cv8c.exe /Execute /C "<сценарий>"`, не через `ФоновыеЗадания.Выполнить`. В клиент-серверной кластере ФЗ могут попадать в очередь одного rphost → последовательное выполнение → нет конкуренции блокировок → 0 TLOCK/TDEADLOCK.
- **Расширение `ТЖМоделированиеРасш` опционально** — не требуется для текущих сценариев (всегда идёт Shell). Оставлено для будущих кейсов, где может понадобиться запуск из ФЗ.
- **Доступ к методам модуля объекта из формы** — только через `РеквизитФормыВЗначение("Объект")`. На клиенте/сервере формы `Объект` — это `ДанныеФормыСтруктура` без методов.
- **CompatibilityMode расширения** — `Version8_3_27` (совпадает с версией платформы). Несовпадение ломает сериализацию `ДинамическийСписок` в формах БСП.

Детали — в [docs/SPRINT_3_5_REPORT.md](../../docs/SPRINT_3_5_REPORT.md), раздел «Решённые архитектурные проблемы».
