### Симптом 1 — предопределённые счета (исправлено в `99923aa`)

**Скриншот юзера:** `SCREENS/2026-05-23_17-41-34.png` (страница «Анализ запроса»,
рядом окно Конфигуратора с планом счетов Хозрасчетный).

**Запрос:**
```sql
ВЫБРАТЬ Хозрасчетный.Ссылка
ИЗ ПланСчетов.Хозрасчетный КАК Хозрасчетный
ГДЕ Хозрасчетный.Ссылка В ИЕРАРХИИ (
    ЗНАЧЕНИЕ(ПланСчетов.Хозрасчетный.РасчетыСПрочимиПокупателямиИЗаказчиками),
    ЗНАЧЕНИЕ(ПланСчетов.Хозрасчетный.РасчетыСПрочимиПокупателямиИЗаказчикамиВал),
    ЗНАЧЕНИЕ(ПланСчетов.Хозрасчетный.РасчетыСПрочимиПокупателямиИЗаказчикамиУЕ)
)
```

**Что выдавал валидатор:** 3 × «КРИТИЧНО · Реквизит "РасчетыСПрочимиПокупателямиИЗаказчиками…"

**Почему было неправильно:** имена внутри `ЗНАЧЕНИЕ()` — это
**ПРЕДОПРЕДЕЛЁННЫЕ счета** из `Predefined.xml`, а не реквизиты плана счетов.

**Что было исправлено в `99923aa`:**
1. `check_attribute_not_exists_in_from_alias` теперь маскирует содержимое
   `ЗНАЧЕНИЕ(...)` пробелами перед поиском alias.X. Также в lookbehind regex
   добавлена `.` — чтобы `Alias.X` не матчился в составе `Kind.Alias.X`.
2. Добавлен парсер `Ext/Predefined.xml` (рекурсивно через `<ChildItems>`),
   таблица `predefined_items` в SQLite (`SCHEMA_VERSION` 1 → 2 с авто-сбросом
   кеша).
3. Новое правило `predefined_item_not_exists` — ловит typo в имени
   предопределённого, использует Levenshtein для подсказок похожих.

### Симптом 2 — `ПустаяСсылка` (стало триггером для скрытия раздела)

**Скриншот юзера:** `SCREENS/2026-05-23_17-46-24.png` (но скрин был отправлен
в чат, не в SCREENS — см. transcript).

**Запрос (фрагмент):**
```sql
И СписаниеСРасчетногоСчетаРасшифровкаПлатежа.СтатьяДвиженияДенежныхСредств.ВидДвиженияДенежныхСредств <>
    ЗНАЧЕНИЕ(Перечисление.ВидыДвиженийДенежныхСредств.ПустаяСсылка)
```

**Что выдавал валидатор:** «КРИТИЧНО · Значение "ПустаяСсылка" не существует

**Почему было неправильно:** `ПустаяСсылка` — это **специальный конструктор**,
доступный у **любого** ссылочного типа:
- `Справочник.Х.ПустаяСсылка`
- `Документ.Х.ПустаяСсылка`
- `Перечисление.Х.ПустаяСсылка`
- `ПланСчетов.Х.ПустаяСсылка`
- `ПланВидовХарактеристик.Х.ПустаяСсылка`
- `ПланВидовРасчета.Х.ПустаяСсылка`
- `БизнесПроцесс.Х.ПустаяСсылка`
- `Задача.Х.ПустаяСсылка`

То есть это не значение перечисления и не предопределённый элемент. Это
синтаксическая конструкция языка запросов 1С. Текущие чекеры её не понимают.

### Что ещё сломано (не проверено, но архитектурно неизбежно)

Юзер не проверил все случаи — потому и был принят principle of caution. По
коду semantic_checks.py видны минимум следующие классы проблем, которые
скорее всего тоже даёт FP:

#### A. Стандартные реквизиты по типу объекта
`check_attribute_not_exists_in_from_alias` имеет хардкод-список из ~20
стандартных реквизитов (`Ссылка`, `Код`, `Наименование`, `ПометкаУдаления`,
`Дата`, `Номер`, `Проведен`, …). Реально стандартные реквизиты сильно
различаются по типу:

| Тип объекта | Специфичные стандартные реквизиты |
|---|---|
| Документ | МоментВремени, ВерсияДанных, ПроведенЗакрытым |
| Регистр накопления | ВидДвижения, Регистратор, Период, Активность, НомерСтроки |
| Регистр сведений | Период, Регистратор, Активность |
| Регистр бухгалтерии | Счет, Субконто1..3 (или сколько настроено), СчетДт, СчетКт, СубконтоДт1..3, СубконтоКт1..3, ВидСубконто1..3, Сумма, СуммаДт/Кт, Регистратор, Период |
| Регистр расчёта | ПериодДействияНачало, ПериодДействияКонец, ПериодРегистрации, БазовыйПериодНачало, БазовыйПериодКонец, ВидРасчета |
| План счетов | Порядок, Вид, Забалансовый, ВидыСубконто (табчасть), Признаки (табчасть) |
| Бизнес-процесс | Завершен, ВедущаяЗадача, СтартованОбъектом |
| Задача | БизнесПроцесс, ТочкаМаршрута, Выполнена, Дата, Наименование |
| Журнал документов | Регистратор, Дата, Номер, тип Документа (Тип) |

Хардкод не покрывает почти ничего из этого. Любой запрос к регистру
бухгалтерии (`РегистрБухгалтерии.Хозрасчетный КАК Х … Х.Счет, Х.СуммаДт`)
сразу даст 2-3 FP на стандартных реквизитах регистра.

#### B. FROM на виртуальную таблицу регистра
```sql
ВЫБРАТЬ Т.Склад, Т.Номенклатура, Т.КоличествоОстаток
ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(&Дата, ) КАК Т
```

У `Т` появляются поля: измерения (Склад, Номенклатура), ресурсы с суффиксом
Остаток/Приход/Расход/Оборот (КоличествоОстаток, КоличествоПриход…). Текущий
код в `check_attribute_not_exists_in_from_alias` вообще не находит эту
конструкцию через свой `from_re` (regex охватывает только статичные типы:
Справочник, Документ, ПланСчетов, ПланВидовХарактеристик, ПланВидовРасчета,
ЖурналДокументов). На VT-таблицах валидация просто отсутствует — но если
кто-нибудь дополнит regex, FP начнут сыпаться, потому что виртуальные имена
полей не известны парсеру.

#### C. FROM на табчасть документа
```sql
ВЫБРАТЬ Т.Номенклатура, Т.Количество, Т.Ссылка.Дата
ИЗ Документ.ПоступлениеТоваровУслуг.Товары КАК Т
```

У `Т`:
- Поля из табчасти `Товары` (Номенклатура, Количество, Цена, …)
- Поле `Ссылка` ведёт на сам документ — через него доступны все реквизиты
  шапки (Дата, Номер, Контрагент, …)
- Поле `НомерСтроки`

Это **не** Документ.ПоступлениеТоваровУслуг — это его табчасть. Текущий код
этого не понимает.

#### D. Цепочки `.Поле.Поле.Поле`
```sql
ВЫБРАТЬ К.Договор.Контрагент.ИНН
ИЗ Справочник.КонтрагентскиеДоговоры КАК К
```

Чтобы валидировать `.ИНН`, нужно понимать, что `.Контрагент` — это
ссылочный реквизит типа `СправочникСсылка.Контрагенты`, и тогда `.ИНН`
проверяется уже у `Справочник.Контрагенты`, а не у `КонтрагентскиеДоговоры`.

Текущий код пытается найти `.ИНН` среди реквизитов `КонтрагентскиеДоговоры`
— и выдаёт FP. Без type chasing это принципиально не лечится.

#### E. Подзапросы и временные таблицы
```sql
ВЫБРАТЬ * ИЗ (ВЫБРАТЬ Х.Ссылка КАК Ссылка ИЗ Справочник.Контрагенты КАК Х) КАК Подзапрос
```

У `Подзапрос.Ссылка` — это поле, выбранное в подзапросе, не имеет отношения
к Контрагенты. Текущий код этого не понимает.

```sql
ВЫБРАТЬ * ИЗ ВТ_МойРезультат КАК ВТ
```

`ВТ_МойРезультат` — временная таблица, у неё свой набор полей из ПОМЕСТИТЬ-блока,
который не в конфигурации. Текущий код silently skip-ает потому что
`is_object_exists("ВТ_МойРезультат")` = false, но это случайность, а не
правильная логика.

## Архитектурная причина: regex ≠ грамматика SDBL

Современный код в `semantic_checks.py` — **набор regex-ов**, каждый ищет
паттерн в плоском тексте запроса. Это:

- Не знает scope.
- Не знает типы.
- Не различает строковый литерал и ключевое слово (хотя в SDBL литералы редки).
- Не различает SELECT-list и WHERE-list.
- Не понимает алиасы из подзапросов отдельно от FROM верхнего уровня.
- Не понимает `ПОМЕСТИТЬ ВТ_X` и последующие ссылки на него.

Patches вида «whitelist для ПустаяСсылка» закрывают конкретный паттерн, но
**не устраняют корневую причину**. Каждое следующее обращение пользователя
с реальной БП 3.0 будет находить новый паттерн, который требует ещё одного
whitelist или ещё одной regex-надстройки. Это негодная стратегия для
production-фичи: precision никогда не достигнет 100%.

Пользователь сформулировал критерий явно (2026-05-23 18:18):
> «Точная валидация без ошибок» — либо так, либо фича скрыта.

Поэтому — скрыта.

## Sprint 6: что нужно сделать, чтобы вернуть фичу

### 1. Полноценный SDBL parser (обязательно)

PyParsing-грамматика или ручной recursive-descent. На входе — текст
запроса, на выходе — AST с узлами:

- `Query` (root)
- `Select` (`ВЫБРАТЬ ...`)
- `From` (`ИЗ <source>`)
- `Source` = `ObjectRef` | `VirtualTable` | `TabularSection` | `Subquery` | `TempTable`
- `Join` (`СОЕДИНЕНИЕ ... ПО ...`)
- `Where`
- `GroupBy`, `Having`, `OrderBy`
- `Union` (`ОБЪЕДИНИТЬ` / `ОБЪЕДИНИТЬ ВСЕ`)
- `Expression`: `BinaryOp`, `FunctionCall`, `Case`, `Cast` (`ВЫРАЗИТЬ`),
  `ValueLiteral` (`ЗНАЧЕНИЕ`), `Identifier`, `FieldAccess`
- `Identifier` с разрешением через scope tracker (см. ниже)

Подсказки:
- Грамматика 1С запросного языка хорошо документирована в Синтакс-помощнике
  («Язык запросов» → «Описание языка запросов»).
- BSL Language Server имеет SDBL grammar в Java — можно посмотреть как референс.
- Но **не** интегрировать LSP как зависимость — runtime Java в Tauri 2 не
  тянем. Реализовать парсер на Python.

### 2. Scope tracker (обязательно)

В каждой точке AST знать:
- Какие alias'ы определены сейчас (FROM + JOIN + SUBQUERY).
- Какой объект конфигурации (или подзапрос/ВТ) стоит за каждым alias.
- Для VT — какие поля у этой VT (измерения + ресурсы с суффиксами Остаток/
  Приход/Расход/Оборот).
- Для табчасти — поля табчасти + `Ссылка` + `НомерСтроки`.
- Для подзапроса — список полей из SELECT-list.
- Для ВТ из `ПОМЕСТИТЬ` — список полей из SELECT-list соответствующего
  ПОМЕСТИТЬ-блока.

Реализация: visitor по AST с push/pop scope.

### 3. Type chaser (обязательно)

Для каждого реквизита-ссылки знать **целевой тип** — он есть в
`attributes.type_repr` (например `cfg:CatalogRef.Контрагенты` или
составной `cfg:CatalogRef.X | cfg:CatalogRef.Y`). Когда видим
`alias.Поле1.Поле2.Поле3`, идём по цепочке:
- `Поле1` ищется у объекта, на который указывает `alias`.
- Тип `Поле1` определяет, где искать `Поле2`.
- Аналогично для `Поле3`.

Если тип составной — `Поле2` должно существовать у **всех** компонентов
(иначе finding «реквизит существует не у всех типов составного реквизита,
нужен ВЫРАЗИТЬ»).

### 4. Whitelist специальных конструкций

- `ПустаяСсылка` — доступна у любого ссылочного типа.
- `Тип(<имя>)` — конструктор типа.
- Псевдо-функции: `КОЛИЧЕСТВО`, `СУММА`, `СРЕДНЕЕ`, `МИНИМУМ`, `МАКСИМУМ`,
  `НАЧАЛОПЕРИОДА`, `КОНЕЦПЕРИОДА`, `ДОБАВИТЬКДАТЕ`, `РАЗНОСТЬДАТ`,
  `ГОД`/`КВАРТАЛ`/`МЕСЯЦ`/`ДЕНЬ`/`НЕДЕЛЯ`/`ЧАС`/…, `ПОДСТРОКА`, `ДЛИНА`,
  `ВЫБОР … КОГДА … ТОГДА … ИНАЧЕ … КОНЕЦ`, `ЕСТЬNULL`, `ЕСТЬNULL`,
  `ПРЕДСТАВЛЕНИЕ`, `ССЫЛКА`, `ВЫРАЗИТЬ`, `ЗНАЧЕНИЕ`.
- Стандартные псевдо-поля: `.Тип`, `.Метаданные`, `.УникальныйИдентификатор`
  (но **не** `.Дата` — это реквизит документа, его проверяем как обычно).

### 5. Standard attributes per kind

См. таблицу в разделе «Что ещё сломано», подраздел A. Нужна функция

## Открытые вопросы для архитектора

1. **Pyparsing vs custom recursive-descent.** Лучше PyParsing (быстрее на
   разработку, медленнее на runtime), или ручной парсер на стандартной
   библиотеке? У БП 3.0 запросы бывают до ~2000 строк — производительность
   парсера имеет значение.

2. **Сколько уровней type chasing поддерживать?** В типовой БП 3.0
   встречаются цепочки до 5-6 `.Поле`. Если ограничить тремя — упростит
   реализацию, но даст FP на длинных цепочках. Если делать без ограничения
   — нужно защититься от циклов (составные типы могут вернуть нас в
   исходный объект).

3. **Реакция на отсутствие подключённой конфигурации.** Сейчас все semantic
   rules silent skip-аются. В Sprint 6 если рулы станут точными — может,
   стоит показывать **синтаксические** проверки (синтаксис SDBL + best
   practices ЦУП), даже без конфы, а семантические только при подключении?
   Это уже есть, но UX отделения двух режимов сейчас неявный.

4. **AI-rewriter — что с ним?** Сейчас он работает независимо от
   semantic-валидатора (через Claude API). Скрытие фичи означает, что и
   AI-rewriter недоступен — это **отдельное** value, не зависящее от
   точности валидатора. Может, имеет смысл оставить отдельный мини-экран
   «Переписать запрос через AI» без semantic findings? — на усмотрение
   архитектора.
