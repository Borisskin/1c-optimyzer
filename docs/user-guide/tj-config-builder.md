# Конструктор logcfg.xml

**Горячая клавиша:** Ctrl+L  
**Местонахождение:** Sidebar → ПОДГОТОВКА → Конструктор logcfg.xml

---

## Что такое logcfg.xml и зачем он нужен

Технологический журнал 1С (ТЖ) — встроенный инструмент профилирования производительности. Он записывает события (вызовы, SQL-запросы, блокировки, ошибки) в текстовые файлы. Чтобы включить ТЖ, нужно создать файл `logcfg.xml` с нужными настройками и положить его в папку конфигурации сервера 1С.

**Без logcfg.xml нет данных для анализа** — экраны «Медленные запросы», «Блокировки», «Анализ плана» работают только с уже собранными архивами ТЖ.

---

## Как открыть

- Нажмите **Ctrl+L** из любого экрана
- Или кликните «Конструктор logcfg.xml» в левом меню (раздел ПОДГОТОВКА)

Экран открывается **без загруженного архива** — это инструмент подготовки, не анализа.

---

## Быстрый старт: AI-мастер

1. Откройте экран (Ctrl+L)
2. Выберите таб **AI-мастер** (открывается по умолчанию)
3. Опишите проблему свободным текстом, например:
   - *«Тормозит проведение документов в обед»*
   - *«Дедлоки при работе 50 пользователей»*
   - *«1С на PostgreSQL медленно работает с регистрами»*
4. Укажите СУБД если знаете (необязательно)
5. Нажмите **«Сгенерировать настройку»**
6. AI подберёт нужные события с оптимальными порогами и объяснит почему
7. Нажмите **«Применить в конструкторе»** для просмотра деталей
8. Нажмите **«Скачать logcfg.xml»**

## Графический конструктор

Открывается по клику на таб **«Графический конструктор»**. Позволяет точно настроить каждый параметр.

### Шаблоны

Вверху экрана — 6 встроенных шаблонов. Выберите подходящий как отправную точку:

| Шаблон | Когда использовать | Объём |
|--------|-------------------|-------|
| **Минимальный** | Постоянный мониторинг в продуктиве | Очень мало (до 10 МБ/ч) |
| **Медленные операции** | Стандартное расследование торможений | Средний (50–200 МБ/ч) |
| **Полная диагностика** | Глубокое расследование (не более 30–60 мин) | Очень большой (0.5–2 ГБ/ч) |
| **Только блокировки** | Расследование дедлоков и конфликтов | Средний (100–300 МБ/ч) |
| **Аудит (1С:Эксперт)** | Профессиональный аудит производительности | Большой (300–700 МБ/ч) |
| **Baseline перед релизом** | Сравнение «до и после» обновления | Мало (30–100 МБ/ч) |

После выбора шаблона можно изменить любые параметры.

### События

Каждое событие можно включить/выключить. Для некоторых событий доступен порог длительности в **centiseconds** (1 cs = 10 мс):

| Значение | Что означает |
|----------|-------------|
| 100 cs | Только события дольше 1 секунды |
| 10 cs | Только события дольше 100 мс |
| 0 | Все события без фильтра |

Нажмите **?** рядом с именем события чтобы узнать что оно делает.

**Основные события:**

- **EXCP / EXCPCNTX** — ошибки и стек вызовов. Включайте всегда.
- **TDEADLOCK / TTIMEOUT** — дедлоки и таймауты блокировок. Включайте при проблемах с блокировками.
- **CALL** — медленные операции бизнес-логики. Порог 100–1000 cs.
- **SDBL / DBMSSQL / DBPOSTGRS** — SQL-запросы. DBMSSQL для MS SQL Server, DBPOSTGRS для PostgreSQL.

**Дополнительные события** (кнопка «Ещё N события»):

- **SCALL** — серверные вызовы. При лагах интерфейса.
- **TLOCK** — ожидания блокировок. При конфликтах записи.
- **ADMIN** — административные события. При проблемах с сеансами.
- **MEM** — использование памяти. Только при подозрении на утечку.
- **ATTN** — обрывы соединений с СУБД.

### Планы запросов

Переключатель **«Собирать планы запросов»** добавляет в лог планы выполнения SQL (plansqltext). Нужен для последующего анализа в экране «Анализ плана».

**Внимание:** включение планов увеличивает объём логов в 3–4 раза.

### Хранилище

- **Папка логов** — куда 1С пишет файлы ТЖ. По умолчанию `C:\1C-TechLog`. Убедитесь что путь существует и есть место на диске.
- **Лимит (ГБ)** — при превышении старые файлы удаляются автоматически. История хранится 24 часа.

### Ожидаемый объём

Карточка справа показывает приближённую оценку объёма для трёх уровней нагрузки:
- **Тихая нагрузка** — ~10 пользователей
- **Типичная** — ~50 пользователей
- **Пиковая** — 200+ пользователей

Оценка приблизительная и может отличаться от реальной в 2–5 раз.

## После скачивания logcfg.xml

1. **Скопируйте файл** в папку `conf\` на сервере 1С (рядом с `ragent.exe`).
   Обычно это `C:\Program Files\1cv8\<версия>\bin\conf\` или `C:\Program Files (x86)\1cv8\<версия>\bin\conf\`.
2. **Перезапустите службу** «1C:Enterprise Server Agent» в Диспетчере служб Windows.
3. **Воспроизведите проблему** — проведите 10–30 минут в системе с типичной нагрузкой.
4. **Остановите сбор** — удалите logcfg.xml и снова перезапустите агент.
5. **Загрузите архив** — откройте папку с логами в Optimyzer (кнопка «Открыть архив» или перетащите папку).

## Часто задаваемые вопросы

**Q: Куда именно класть logcfg.xml?**  
В папку `conf\` рядом с исполняемыми файлами сервера. Если 1С установлен в нестандартную папку — путь совпадает с расположением `ragent.exe`.

**Q: Нужно ли останавливать 1С на время настройки?**  
Нет — достаточно перезапустить только службу «1C:Enterprise Server Agent». Рабочие процессы перезапустятся автоматически.

**Q: Почему объём логов больше ожидаемого?**  
Оценка в конструкторе приблизительная. На реальном объёме влияют: число активных пользователей, частота обращений к базе, сложность запросов. При слишком большом объёме увеличьте пороги событий или уменьшите набор событий.

**Q: Что если AI вернул непонятную настройку?**  
Переключитесь на таб «Графический конструктор» (кнопка «Применить в конструкторе») — там можно детально просмотреть и изменить каждое событие.

**Q: Можно ли собирать ТЖ только для одной базы?**  
Текущая версия конструктора настраивает сбор для всех баз на сервере. Настройка по конкретной базе — в планах на Sprint 11.
