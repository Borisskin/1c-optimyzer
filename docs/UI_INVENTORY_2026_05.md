## Update 2026-05-23 17:50 — Query Analyzer скрыт до Sprint 6

После UX-сессии Сергей нашёл два паттерна false positive в semantic-валидаторе:

1. `ЗНАЧЕНИЕ(ПланСчетов.Хозрасчетный.X)` — X считался реквизитом, хотя это
   предопределённый счёт (исправлено в `99923aa`).
2. `ЗНАЧЕНИЕ(Перечисление.X.ПустаяСсылка)` — ПустаяСсылка трактовалась как
   значение перечисления, хотя это **специальный конструктор**, доступный у
   любого ссылочного типа (Справочник / Документ / Перечисление / ПланСчетов /
   ПланВидов* / БизнесПроцесс / Задача).

Эти два — лишь верхушка айсберга. Текущий regex-based валидатор имеет
**фундаментальные ограничения**, которые нельзя обойти patches'ами:

- Нет понимания **scope** подзапросов и временных таблиц.
- Нет обработки FROM на виртуальной таблице (`ИЗ РегНак.X.Остатки КАК Y`).
- Нет обработки FROM на табчасть документа (`ИЗ Документ.X.Товары КАК Т`).
- Нет **type chasing** для цепочек `К.Договор.Контрагент.ИНН` (нужно знать тип
  `.Договор` чтобы понимать, какие у него поля).
- Хардкод-список стандартных реквизитов (Ссылка/Код/Наименование/…) не покрывает
  специфичные для типа: РегистрБухгалтерии (Счет, Субконто1..3, СчетДт, СчетКт),
  ПланСчетов (Порядок, Вид, Забалансовый, ВидыСубконто, Признаки), Документ
  (МоментВремени), и т.д.
- Не понимает специальные конструкторы: `Тип(...)`, `ПустаяСсылка`,
  `<Объект>.НайтиПо…` (последнее в SDBL не используется, но псевдо-методы есть).

**Решение Сергея (2026-05-23 18:21):** скрыть пункт «Анализ запроса» из Sidebar
до Sprint 6. Лучше отсутствие функции, чем функция с непредсказуемыми false
positives на типовых запросах БП 3.0.

### Что физически сделано

- Пункт `query-analyzer` закомментирован в `frontend/src/components/chrome/nav.ts`
  (паттерн «Sales build»: код остаётся, UI его не видит).
- Шорткат Ctrl+Q отключён в `App.tsx`.
- Импорт `QueryAnalyzerScreen` закомментирован, в `renderScreen` case
  `"query-analyzer"` отдаёт placeholder (если юзер через CommandPalette или
  ручной `setScreen` попадёт на него — увидит нейтральное «раздел временно
  скрыт»).
- Backend RPC (`query_analyzer.*`) и Python-код остаются нетронутыми — будут
  переиспользованы в Sprint 6.
- ConfigurationBadge / Settings → «Подключить конфигурацию» **тоже остаются** —
  индекс конфы пригодится для других фич (Sprint 6 ConfigDrillDown,
  TJ-Context → Module mapping и т.п.).

### Что нужно для Sprint 6 reintroduction

Чтобы вернуть «Анализ запроса» с гарантией 0 false positives:

1. **Полноценный SDBL-парсер.** PyParsing или ручной recursive-descent на грамматику
   1С запросного языка (FROM/JOIN/UNION, подзапросы, временные таблицы, ВЫРАЗИТЬ,
   ЗНАЧЕНИЕ, виртуальные таблицы с параметрами, ИТОГИ ПО, ИНДЕКСИРОВАТЬ ПО,
   ОБЪЕДИНИТЬ ВСЕ, и т.д.).
2. **Scope tracker.** В каждой точке запроса знать, какие алиасы определены и на
   какой реальный объект конфигурации они указывают.
3. **Type chaser.** Для каждого реквизита-ссылки знать тип — чтобы цепочка
   `К.Договор.Контрагент.ИНН` валидировалась как «у объекта типа `Договор`
   найти реквизит `Контрагент` → у объекта типа `Контрагент` найти реквизит
   `ИНН`».
4. **Whitelist специальных конструкторов:** ПустаяСсылка (для всех ссылочных),
   `Тип(...)`, `СписокЗначений`, `КОЛИЧЕСТВО(*)`, `НАЧАЛОПЕРИОДА`, и т.д.
5. **Standard-attributes per-kind:** определять полный набор стандартных
   реквизитов по типу объекта (Документ vs Справочник vs Регистр …), а не
   единый хардкод-сет.
6. **Acceptance gate:** прогнать 20-30 реальных запросов из типовой БП 3.0
   (см. подсказку Сергея от 2026-05-23 в чате — есть подборка по
   `Documents\ФормированиеЗаписейКнигиПокупок` и `Reports\ЭкспрессПроверка`).
   Только если **0 false positives** на этих запросах — можно возвращать в UI.

См. также: коммит, скрывший раздел (TBA после commit), и `nav.ts` с одной
закомментированной строкой — вернуть feature будет на 1 строку кода (плюс
условия выше).

## Активные экраны — детальное описание

### 1. Бизнес-операции — `Operations.tsx`

Топ-N операций по `context_normalized`, сортировка по `total_duration_ms DESC`.
Колонки: # / Операция / Вызовов / Σ ms / avg ms / max ms / SQL ms / Locks /
EXCP / Сессий. Цветные точки (hot/warm) показывают долю операции в Σ
длительности. Клик по строке → drill-down `Anatomy`. Лимит выборки
конфигурируется через `LimitSelector` (по умолчанию 500). Экспорт CSV/JSON.

### 2. SQL Console — `SQLConsole.tsx`

Полноценный редактор SQL поверх per-archive DuckDB. Слева — schema-панель
(колонки таблицы `events`, клик вставляет имя в редактор), снизу — шаблоны
запросов, справа — сохранённые запросы. Результаты во вкладках «Таблица» и
«Сырой JSON». Поддержка `Ctrl+Enter` для запуска, `Ctrl+/` для комментария.
Видно поле `executed_in`, `scanned_events`, признак `truncated`.

### 3. Медленные запросы — `SlowQueries.tsx`

Топ DBMSSQL событий, сгруппированных по `sql_text_hash` с агрегатами
(`calls`, `total_duration_ms`, `avg`, `max`, `rows_read`, `unique_sessions`).
Сортировка по Σ длительности. Expand-by-click раскрывает строку и показывает
текст запроса, отформатированный через `utils/sqlFormat.ts` (читаемые
переносы по SELECT/FROM/WHERE/JOIN, отступы по скобкам).

### 4. Блокировки — `LocksTimeline.tsx`

LineChart по двум сериям — `locks` (TLOCK) и `deadlocks` (TDEADLOCK) — в
bucket'ах времени (bucket рассчитывается backend'ом исходя из диапазона
архива). Над графиком — суммарные счётчики, экспорт CSV/JSON.

### 5. Роли процессов — `ProcessRoles.tsx`

Donut-диаграмма распределения событий по `process_role` (rphost / rmngr /
ragent / 1cv8 …) + таблица. Поддерживается фильтр-поиск, сортировка по
колонкам, экспорт.

### 6. Длительности — `DurationHistogram.tsx`

Гистограмма событий по бакетам длительности (от микросекунд до часов),
логарифмическая Y-шкала. Над графиком — суммарное число событий, экспорт.

### 7. События ТЖ — `ErrorsFeed.tsx`

Лента событий, не только EXCP. Серверная фильтрация по `event_type` (компонент
`EventTypeFilter`) и по наличию `context` (компонент `ContextFilter`:
любое / есть / нет). LimitSelector (по умолчанию 500). Expand-by-click
раскрывает `extra` JSON.

### 8. Активность — `ActivityHeatmap.tsx`

Тепловая карта 7×24 (день недели × час). Переключатель метрики:
«Количество событий», «Σ длительность», «Пик длительности», «Количество ошибок».
Клик по ячейке выставляет cross-filter по времени.

### 9. Анализ запроса — `QueryAnalyzer.tsx`

Standalone-экран, не требует подключённого архива (но при подключённой
конфигурации — выполняет семантические проверки). Поля: текст 1С-запроса
(SDBL), кнопки «Анализировать», «Очистить», «Переписать через AI». Результат —
список `Findings` по категориям (производительность / корректность / стиль) и
по severity (critical / warning / info). Дополнительно — `ConfigurationBadge`
сверху, диалог подключения XML-выгрузки конфигурации, кнопка AI-rewrite с
показом diff (original / rewritten) и notes от модели.

### 10. Сравнение архивов — `ArchiveComparison.tsx`

Slot A vs Slot B, два селектора архива. Вкладки:
- **Сводка** — diff агрегатных метрик (events_count, Σ длительность, top
  process_role и т.д.).
- **Медленные запросы** — diff топа DBMSSQL: что добавилось / убавилось /
  изменилось.

## i18n — состояние строк

Файл: `frontend/src/i18n/ru.ts`.

### Что переведено

Все 24 видимых пункта Sidebar имеют русские строки в `t.sidebar.items.*`
(см. `ru.ts:62-89`). Все групповые заголовки переведены (`t.sidebar.groups.*`).

Все 9 активных экранов имеют переведённые `breadcrumbs` и `title` (прямо в
JSX каждого экрана, в виде литералов на русском, не через `t.*`).

### Что выглядит placeholder'ом

1. **`t.app.version = "v0.1.0"`** — несоответствие. Проект уже на **v0.5.0**
   (см. `docs/SPRINT_5_REPORT.md` и тег `v0.5.0-internal`). Версия в TopBar и
   StatusBar показывается из этой константы — нужно поднять до 0.5.0.
2. **`t.app.devTag = "dev"`** — захардкожен, не зависит от build mode.
3. **`t.app.screenPlaceholder = 'Экран "{id}" появится в Module 2+'`** —
   fallback в `renderScreen()` для неизвестного `ScreenId`. Срабатывает только
   если в `appStore.ScreenId` появится id, которого нет в `App.tsx` switch
   (сейчас не срабатывает).
4. **`t.sql.sprintLabel = "Sprint 2 · Phase B"`** — на странице SQL Console
   рядом с заголовком висит бейдж «Sprint 2 · Phase B». Это внутренний
   маркер фазы разработки — для production-релиза смотрится странно.
5. **`t.sql.badgeFreeTier = "free tier"`** — англоязычный жаргон, идёт
   вразрез с «hardcoded ru-RU UI» (комментарий на строке 1 `ru.ts`).
6. **`t.sql.results.empty.readyHint = "событий · SQL-консоль появится в
   Phase B"`** — устаревший hint, фаза давно сдана.
7. **`t.sql.editor.placeholderDefault`** — внутри тексте `-- Sprint 2 Phase
   B: SQL Engine появится здесь.` Аналогично устаревший комментарий.
8. **`t.sql.results.placeholderChart = "График — Phase C."` и
   `placeholderTimeline = "Хронология — Phase D."`** — обе вкладки заявлены
   через названия Phase, при этом сейчас они даже не показываются (см.
   `SQLConsole.tsx`: `type ResultTab = "table" | "raw"` — Chart и Timeline
   tabs удалены из UI, но строки остались в i18n).
9. **`t.topbar.alertsTooltip = "Алерты — Module 2"`** — указание на
   несуществующий Module 2.
10. **`t.topbar.aiBadge = "AI"`** + **`aiBadgePro = "Pro"`** — английский,
    `Sparkles` иконка, кнопка disabled. Маркетинговая заглушка.
11. **`t.cmdpalette.commands.recentSourcesHint = "Sprint 1"`** — тост при
    клике на «Недавние источники» в Command Palette: показывает «Sprint 1»
    (потому что реально такой команды ещё нет).
12. **`t.cmdpalette.aboutToast = "1C-Optimyzer v0.1.0 · Module 1
    (OptimyzerQL Standalone)"`** — упоминание «Module 1», устаревшая версия.
13. **`t.cmdpalette.hints.file = "Файл · Module 1"`** и
    **`hints.list = "Список · Module 1"`** — то же самое.
