## 2. Инвентаризация `C:\1C-TechLog`

- **Объём:** 27 МБ · **483** файла `.log` · **104 416** событий
- **Структура:** кластерная — `ragent_31352`, `rmngr_19096`, `rphost_23972` + ~190 клиентских папок `1cv8_*` / `1cv8s_*` / `1cv8c_*`. Основной объём: `ragent` 10.4 МБ, `rmngr` 7.6 МБ, `rphost` 4.1 МБ; клиентские — почти все 0–0.8 МБ (многие пустые)
- **Имена файлов:** маска `YYMMDDHH.log` — **483 / 483** совпадают → парсер корректно восстановит timestamp
- **Кодировка:** UTF-8 с BOM (`EF BB BF`) → парсер читает `utf-8-sig`, совместимо
- **Совместимость парсера:** полная (структура `role_pid` + маска + кодировка — всё как ожидает `tj_parser` / `FolderSource`)

## 4. Покрытие сценариев ЭТИМ архивом — главный вывод

Архив собран с настройкой ТЖ, фиксирующей **клиент-серверный обмен + исключения + управляемые блокировки**, но **без слоя СУБД**.

### Покрыто (можно валидировать сразу)
- **Ingest & надёжность** — 483 файла, кластер, пустые файлы, кодировка, скорость, повторная загрузка
- **Лента ошибок** — EXCP 9 779 + EXCPCNTX 520 (богатый материал; заодно регресс fix EXCPCNTX double-count)
- **Timeline блокировок (TLOCK)** — 82 события (но `WaitConnections=0` → блокировки без конфликтов: проверить «пустой конфликт» в UI)
- **CALL-аналитика** — топ бизнес-операций, анатомия операции, роли процессов, heatmap, гистограмма длительностей (вкл. аномалию ~10 ч)
- **Анатомия сессии**, **SQL-консоль** (`execute_sql` по событиям), **пресеты** (`first_100`, `longest`)
- **Регрессии** — сравнение двух прогонов по CALL/EXCP
- **logcfg-конструктор** + детект платформы; **AI logcfg-мастер**, **AI по регрессиям**

### НЕ покрыто этим архивом (нет нужных событий)
| Сценарий | Чего не хватает |
|----------|-----------------|
| Медленные запросы / Top SQL | нет `DBMSSQL`/`Sql=` |
| Антипаттерны MSSQL (9) / PG (15) | нет SQL-текстов |
| Engine dispatch MSSQL vs PG | нет DB-событий |
| Планы «из архива ТЖ» | нет `planSQLText=` |

| Конфликты блокировок | `WaitConnections=0` |
| AI explain_query / explain_plan | нет SQL/планов на вход |

Это **флагманские** сценарии продукта (медленные запросы, планы, антипаттерны) и **главная зона риска AI-галлюцинаций** (1С-специфика: запросы/индексы/блокировки). На текущем архиве их не проверить.

## 6. Метод прогона (предложение)

Готового драйвера в репозитории нет. Предлагаю два слоя:

1. **Headless-драйвер** `scripts/s12_run.py` (напишу в Фазе 1): in-process `FolderSource(C:\1C-TechLog) → DuckDBStore → все view_* / preset / sql_antipatterns` → структурированный отчёт (counts, top-N, аномалии) **без значений полей**. Повторяемо, не трогает UI, быстро на 27 МБ.
2. **UI-прогон через `start.bat`** — для AI-сценариев (идут на cloud-сервер) и визуальной оценки осмысленности/UX владельцем (drag-drop папки `C:\1C-TechLog`).
3. **Планы:** прогон PlanAnalyzer на `fixtures/real_world/mssql_plans/*.sqlplan`.

Побочно: спящий тест `test_real_archive_acceptance` (Sprint 0 DoD #19, сейчас `skip`) можно активировать на этом архиве → закрыть acceptance-gate (данные не коммитим — приватность).
