# Подключение XML-выгрузки конфигурации 1С к Query Analyzer

> **Зачем это нужно:** без подключённой выгрузки Query Analyzer проверяет запрос **только синтаксически** (13 правил по методике ЦУП — индексы, виртуальные таблицы, OR/IN в условиях, и т.п.). С подключённой выгрузкой добавляются **8 семантических правил**, которые сверяют запрос с **реальной структурой вашей конфигурации**:
>
> - Существуют ли объекты, упомянутые в `ИЗ` / `СОЕДИНЕНИЕ`
> - Существуют ли реквизиты у объектов
> - Корректна ли виртуальная таблица для типа регистра
> - Существуют ли значения перечислений
> - И т.д.
>
> Это уникальная фича 1C-Optimyzer — ни ЦУП, ни другие продукты в РФ-сегменте такого не делают.

---

## Шаг 1 — выгрузить конфигурацию из Конфигуратора

1. Откройте свою базу 1С через **Конфигуратор** (не через Предприятие).
2. Меню «**Конфигурация**» → «**Выгрузить конфигурацию в файлы XML…**».
3. Выберите пустую папку (рекомендуется отдельную, например `C:\BUFFER\SCHEME\`).
4. Дождитесь окончания выгрузки. Для типовой конфигурации (БП 3.0, УТ, ERP) это занимает несколько минут.
5. В выбранной папке появятся подпапки `Catalogs/`, `Documents/`, `AccumulationRegisters/`, и т.д., плюс `Configuration.xml` на верхнем уровне.

> **Совет:** делайте выгрузку **в отдельную папку**, не в исходники проекта 1С. После каждого крупного релиза конфигурации перевыгрузите — Optimyzer переиндексируется автоматически по хешу.

## Шаг 2 — подключить выгрузку в Optimyzer

1. Откройте **Анализ запроса** (sidebar «АНАЛИЗ» → «Анализ запроса»).
2. В правом верхнем углу экрана — chip «**Конфигурация не подключена**». Кликните по нему.
3. Откроется диалог «Конфигурация 1С». Нажмите «**Указать папку выгрузки…**».
4. В Tauri folder picker выберите ту же папку, что использовали на шаге 1.
5. Дождитесь toast-уведомления «**Конфигурация подключена: N объектов проиндексировано**». Это занимает 10-30 секунд для типовой конфигурации.

После подключения chip в шапке становится зелёным и показывает имя конфигурации + количество объектов.

## Шаг 3 — анализируйте запросы как обычно

Теперь когда вы вставляете запрос в редактор и нажимаете «Анализировать», помимо синтаксических findings (Sprint 4) — могут появляться **семантические** с категорией `semantic`:

| Rule | Когда срабатывает |
|---|---|
| `object_not_exists` | `ИЗ Справочник.Х` — но `Х` нет в конфигурации |
| `attribute_not_exists_in_from_alias` | `К.НесуществующийРеквизит` через алиас FROM |
| `virtual_table_not_supported` | `РегистрНакопления.Х.СрезПоследних` (vtable для регистров сведений) |
| `vyrazit_type_not_exists` | `ВЫРАЗИТЬ(... КАК Документ.Y)` где Y не существует |
| `enum_value_not_exists` | `Перечисление.Х.НесуществующееЗначение` |
| `register_dimension_or_field_missing` | Параметр виртуальной таблицы — не измерение регистра |
| `register_resource_used_as_dimension` | Фильтр виртуалки по ресурсу, не измерению |
| `constant_used_with_dot` | `Константа.Х.Поле` (у констант нет полей) |

Все findings показываются в правой панели наравне с синтаксическими, с подсветкой проблемного места в редакторе.

## Когда переиндексировать

Каждый раз, когда вы **обновляете конфигурацию** (добавили новый объект / удалили / переименовали), перевыгрузите из Конфигуратора в ту же папку.

Optimyzer вычисляет хеш папки выгрузки (имена + размер + mtime всех XML файлов). При следующем нажатии «Анализировать» проверяется:

- Хеш не изменился → используется существующий индекс (быстро).
- Хеш изменился → запускается re-index автоматически через `configuration.connect` (10-30 секунд).

Также в диалоге «Конфигурация 1С» есть кнопка «**Переиндексировать**» — принудительная переиндексация без изменения папки (полезно если вы переписали XML файлы вручную и хотите чтобы кеш точно обновился).

## Отключение

В диалоге «Конфигурация 1С» нажмите «**Отключить**». Это:

- Очистит SQLite индекс конфигурации в `backend/data/config_metadata.db`.
- НЕ затронет саму папку выгрузки — она остаётся на диске.
- Семантические rules перестанут запускаться. Синтаксические продолжают работать как обычно.

Если позже захотите подключить обратно — можно указать ту же папку через «Указать папку выгрузки…», или, если есть env `OPTIMYZER_CONFIG_DB_PATH` указывающий на сохранённую БД — она подхватится автоматически.

## FAQ

**Q: Какие конфигурации поддерживаются?**
A: Любая конфигурация 1С 8.3+ с выгрузкой через Конфигуратор (формат `MetaDataObject` с пространством имён `http://v8.1c.ru/8.3/MDClasses`). Проверено на БП 3.0 v3.0.39.57 (1647 объектов, парсинг ~10 секунд).

**Q: Можно ли подключить несколько конфигураций одновременно?**
A: Sprint 5 — нет. Один индекс на один tool. Если работаете с двумя базами — переключайте через «Указать папку выгрузки…» / «Отключить».

**Q: Что лежит в `backend/data/config_metadata.db`?**
A: SQLite с таблицами `objects`, `attributes`, `tabular_sections`, `enum_values`, `meta`. Метаданные конфигурации — без бизнес-данных. Можно безопасно удалить — пересоздастся при следующем `configuration.connect`.

**Q: Optimyzer обращается к моей базе 1С?**
A: Нет. Sprint 5 работает только со статической XML-выгрузкой. Никаких подключений к 1С:Предприятие, никаких внешних компонент, никаких сетевых запросов к базе. Sprint 6+ может опционально добавить runtime-интеграцию через MCP BSL Atlas, но это будет включаемая фича.

**Q: Можно ли использовать выгрузку из EDT?**
A: Да, формат идентичен — EDT использует тот же `MetaDataObject` контейнер. Парсер не требует, чтобы выгрузка была именно из Конфигуратора.
