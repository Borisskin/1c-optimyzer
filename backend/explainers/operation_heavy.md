# Тяжёлая операция в архиве

## Что произошло

Операция «{{operation}}» в сумме за наблюдаемый период заняла
{{total_duration_ms}} мс (это {{total_duration_s}} секунд) на {{calls}} вызовов.
Это **один из самых трудозатратных контекстов** в архиве.

## Почему так получилось

Это generic fallback правило — конкретный паттерн (heavy SQL / locks / cascade
calls) **не сработал**, потому что соответствующие условия не были выполнены
точно. Но total time высокий — значит, что-то заслуживает внимания.

## Что делать

1. **Открыть «Анатомия операции»** для этого context'а — там распределение по
   event_type показывает где время уходит:
   - больше всего DBMSSQL → правило `slow_op_heavy_sql`
   - много TLOCK/TDEADLOCK → правило `slow_op_locks`
   - много CALL/SCALL с маленьким avg → правило `slow_op_call_cascade`

2. **Использовать AI explainer** (кнопка рядом с rule explanation) — он
   рассмотрит конкретный контекст и предложит более точный анализ.
