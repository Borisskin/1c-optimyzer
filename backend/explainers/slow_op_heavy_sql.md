# Тяжёлый SQL — основная причина торможения

## Что произошло

Операция «{{operation}}» в среднем тратит {{avg_duration_ms}} мс на вызов,
из которых **{{sql_share_pct}}% времени** уходит на SQL-запросы к СУБД
(в сумме {{sql_duration_ms}} мс на {{calls}} вызовов).

Это означает: большая часть времени тратится **не на бизнес-логику в платформе,
а на ожидание ответа от СУБД**.

## Почему так получилось

Типичные причины:
- **Полное сканирование таблицы** вместо использования индекса (нет индекса по
  условию, или статистика устарела и оптимизатор не выбрал индекс).
- **Подзапрос в JOIN** — частая ошибка в 1С-запросах, особенно с виртуальными
  таблицами регистров (Остатки/Обороты).
- **Большой объём данных в выборке** — `ВЫБРАТЬ * ИЗ` без условий, или с
  условием по неиндексированному полю.
- **Расчёт виртуальной таблицы Остатки** на крупном регистре без оптимизации
  итогов (`УстановитьПериодРассчитанныхИтогов`).

## Что делать

1. **Открыть «Медленные запросы»** (быстро). В этом инструменте найдите топ SQL
   именно этой операции — по совокупной длительности или по количеству вызовов.

2. **Проверить индексы** (среднее). Часто оптимизация запроса 1С = добавить
   индекс СУБД по условию WHERE/JOIN. Платформа сама создаёт индексы по
   измерениям регистров, но не по «свободным» полям.

3. **Переписать тяжёлый запрос** (долго). Используйте `ВЫРАЗИТЬ КАК`,
   `ОБЪЕДИНИТЬ ВСЕ` вместо `ОБЪЕДИНИТЬ`, временные таблицы для разбивки
   на этапы. Подзапросы в JOIN — переносите во временные таблицы.
