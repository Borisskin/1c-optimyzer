# Оператор НЕ В с подзапросом

## Что не так

`НЕ В (ВЫБРАТЬ ...)` — антипаттерн почти всегда. Это **anti-semi-join**, который у оптимизатора СУБД ассоциируется с full scan + NULL-handling overhead.

## Почему это плохо

- Семантика `НЕ В` сложнее чем `В`: если в подзапросе есть `NULL`, результат внешнего запроса будет пуст (3VL логика SQL).
- Большинство оптимизаторов СУБД генерируют для `NOT IN` план через `LEFT ANTI JOIN` или `NOT EXISTS`, но не всегда удачно.
- Подзапрос внутри `НЕ В` без индекса работает как O(N²).

## Как переписать

### Замена на ЛЕВОЕ СОЕДИНЕНИЕ + IS NULL

```sdbl
-- Плохо:
ВЫБРАТЬ ...
ИЗ Справочник.Номенклатура
ГДЕ Ссылка НЕ В (ВЫБРАТЬ Номенклатура ИЗ РегистрСведений.Цены)

-- Лучше:
ВЫБРАТЬ Ном.*
ИЗ Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.Цены КАК Цены
        ПО Ном.Ссылка = Цены.Номенклатура
ГДЕ Цены.Номенклатура ЕСТЬ NULL
```

### С временной таблицей если подзапрос сложный

```sdbl
ВЫБРАТЬ Номенклатура ПОМЕСТИТЬ ВТ_ЕстьЦена ИЗ РегистрСведений.Цены
ИНДЕКСИРОВАТЬ ПО Номенклатура;

ВЫБРАТЬ Ном.*
ИЗ Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ ВТ_ЕстьЦена КАК ЕстьЦена
        ПО Ном.Ссылка = ЕстьЦена.Номенклатура
ГДЕ ЕстьЦена.Номенклатура ЕСТЬ NULL
```
